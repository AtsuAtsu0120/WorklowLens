# WorkflowLens プロジェクト

ゲーム開発効率化ツールのワークフローを可視化・管理するためのOSSロギング基盤。
ゲーム開発現場で使われる各種ツール（エディタ拡張、ビルドパイプライン、アセット管理など）の操作ログを収集・転送・閲覧し、開発プロセスの改善に役立てることを目的とする。

## 仕様書駆動設計 (SDD)

このプロジェクトは**仕様書駆動設計（Spec-Driven Design）**を前提とする。

### 基本ルール

1. **実装前に仕様書を確認する** — 機能の実装・修正を依頼されたら、まず `Document/` 配下の該当する仕様書を読む
2. **仕様書がなければ先に作成する** — 該当する仕様書が存在しない場合、実装に入る前に仕様書を作成し、ユーザーの承認を得る
3. **実装後に仕様書を更新する** — 実装の結果、仕様と差異が生じた場合は仕様書を更新する
4. **仕様書が正（Single Source of Truth）** — コードと仕様書に矛盾がある場合、仕様書に合わせるのが原則。仕様書側を変えるべき場合はユーザーに確認する

### ディレクトリ構成

```
各サブプロジェクト/
  Document/
    overview.md          全体概要・アーキテクチャ
    features/            機能別の仕様書
```

### 仕様書フォーマット

- Markdown形式
- 日本語で記述
- 機能単位で1ファイル（`Document/features/` 配下）

## プロジェクト構成

- `workflow_lens_middleware/` — UDP受信→HTTP転送の中継サーバー（Go）
- `workflow_lens_server/` — ログ保存・閲覧のWebサーバー（Go）
- `workflow_lens_client_csharp/` — UDP送信クライアントライブラリ（C# / .NET Standard 2.1）
- `workflow_lens_client_python/` — UDP送信クライアントライブラリ（Python 3.7+）

## 共通規約

- コメント・ドキュメントは日本語
- ユーザーはC#に精通、Goもある程度わかる。冗長な言語基礎の説明は不要

## 作業時の確認ルール

- **判断に迷ったら必ず `AskUserQuestion` ツールを使ってユーザーに確認する**
- 特に以下の場面では勝手に進めず確認すること：
  - 実装方針が複数ありえるとき
  - 仕様書に記載がない挙動を決める必要があるとき
  - 既存コードの変更範囲が広くなりそうなとき
  - 要件が曖昧・不明確なとき
- 自分で推測して進めるより、一度聞いてから進めるほうが望ましい
